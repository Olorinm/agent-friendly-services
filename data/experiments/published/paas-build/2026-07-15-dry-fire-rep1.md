## Summary

**Provider:** paas.build (a payment-facilitator built on UniPaaS, FCA-authorised Payment Institution No. 929994).

**Docs reviewed (official only):** `https://paas.build/agents`, `https://paas.build/llms.txt`, `https://paas.build/SKILL.md`, `https://paas.build/llms-full.txt`, `https://github.com/UNIPaaS/paas-build-mcp` (read raw `paas-build-mcp.mjs` source).

**Canonical API shape:** No traditional API-key + `Authorization` header scheme, and no "list resources" GET endpoint exists. The documented flow is three POST calls: `identify_business` → `go_live` (creates a **real** merchant account, forbidden here) → `create_checkout`. The credential is a `vendorId` (returned only by `go_live`), passed in the JSON body — confirmed by reading the actual MCP server source, which sends no `Authorization` header at all.

Since I cannot call `go_live` (hard rule against account creation/signup), I used the documented `create_checkout` endpoint (`POST https://paas.build/api/checkout`) with an obviously-fake `vendorId`, per task step 2.

**Result:** Outer HTTP status was `502`, but the JSON body deterministically embeds a genuine upstream `401 Unauthorized` (`{"statusCode":401,"message":"Unauthorized","code":401}`), reproduced identically across two different fake IDs — and clearly distinct from the `400 {"error":"vendorId and amount required"}` returned when `vendorId` is omitted entirely. This proves the endpoint, method, headers, and request shape are correct, and that the backend specifically rejected the fake credential (not a generic routing failure). No official doc specifies an error-response shape at all, so I graded this a **pass at medium confidence** — the 502 envelope is undocumented and non-standard, which is the main caveat.
